## üêû Known issues (reported this session)

### Issue 1 ‚Äî Group messages don't reach the backend (local echo only)
**Status:** confirmed in code. **Severity:** high (blocks group chat).
**Root cause:** `Models/Stores.swift` ‚Üí `ChatStore.send(...)`:
```swift
guard let conv = directConversations.first(where: { $0.id == conversationId }) else {
    // Group (or unknown) ‚Äî local echo only for now; mark sent.
    markStatus(tempId, in: conversationId, to: .sent)
    return            // ‚Üê never calls ChatEngine.sendText / the backend
}
```
A group conversation is not in `directConversations`, so the send hits this guard,
shows a local "sent" echo, and **returns without ever calling the backend**. The
message looks sent but is never relayed. This is intentional-but-incomplete: real
group E2E needs the **OpenMLS / GroupSession** path wired (the bigger lift noted
in the original plan). Until then, groups create + show members, but messaging is
local-only.
**Fix path:** wire group sends through the MLS `GroupSession` in e2e-core (encrypt
to the group, fan out via `/messages/send` per the group's conversation_id), and
make the receive path decrypt group messages. Direct (1:1) sends already work and
DO hit the backend.

### Issue 2 ‚Äî Images don't send (placeholder text only)
**Status:** confirmed in code. **Severity:** high (blocks media).
**Root cause:** `Main/ChatDetailView.swift` photo picker:
```swift
.onChange(of: photoItem) { _, item in
    Task {
        if let data = try? await item?.loadTransferable(type: Data.self) {
            chat.send("üì∑ Photo", kind: .image, to: conversation.id)
            photoItem = nil; _ = data   // ‚Üê `data` is loaded then DISCARDED
        }
    }
}
```
The image bytes are loaded but thrown away; it sends the literal text "üì∑ Photo".
The new `ChatEngine.sendMedia(data:mime:...)` (built this session) is **not wired**
to this picker yet. Voice notes have the same shape (line ~420 sends
`"üé§ Voice message ¬∑ Ns"` text, no audio).
**Fix path (small, no new infra ‚Äî backend + engine already done):** in the picker
`onChange`, call `ChatEngine.shared.sendMedia(data, mime: "image/jpeg",
conversationId:, peerUserId:)` instead of `chat.send("üì∑ Photo", ...)`; render
received media by calling `ChatEngine.fetchMedia(ref)` for messages whose
`media != nil`. **Requires** the R2 env on the box first (see "Left" below) so
`/media/*` returns 200 rather than 503.

### Issue 3 ‚Äî Contacts not appearing in NewChat search (but DO appear in NewGroup)
**Status:** needs live repro. **Severity:** medium.
**Observation:** both `NewChatView` and `NewGroupView` read the same
`ContactsService.shared.discover().matches`, and you saw contacts appear in the
group picker but not in the 1:1 "new chat" search. Both filter by
`displayName.localizedCaseInsensitiveContains(search)`. Because the data source is
identical, the most likely causes (need a logged-in device to confirm ‚Äî couldn't
repro here without auth):
  1. **Timing / empty-on-first-open:** `NewChatView.load()` is async; if the
     `.searchable` field is typed into before discovery finishes, `matches` is
     still empty and "No contacts found" shows. (NewGroup shows them after load
     too ‚Äî the difference may just be when you looked.)
  2. **Search text vs saved name:** the filter matches `displayName` only (not
     phone number) for VOIID matches; a contact saved under a different name than
     typed won't match.
  3. **`onVoiid` section hidden when filtered list is empty** ‚Äî if discovery
     returned 0 matches for this account, the "On VOIID" section is omitted
     entirely while invites still render.
**Next step:** reproduce on a logged-in build, log `matches.count` after
`load()` in `NewChatView`, and confirm whether it's empty (discovery/permission
issue) or non-empty-but-filtered (search-string issue). If discovery itself
returns 0, check Contacts permission + `normalizeE164` (defaults to +91; numbers
in other formats won't hash-match).
